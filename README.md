# Manjaro Care

مركز صيانة رسومي لمانجارو/آرتش، بفلسفة نقرة-بدل-أمر — يهدف لإعطاء
مستخدمي لينكس القادمين من Windows تجربة مشابهة لأدوات مثل TuneUp
Utilities أو Advanced SystemCare، لكنها مبنية خصيصاً حول أدوات
النظام الحقيقية (pacman, systemd, journalctl...) بدل منطق مغلق.

## الفلسفة

كل وحدة صيانة تلتزم بثلاث خطوات إلزامية:

1. **فحص (scan)** — قراءة فقط، لا يُغيَّر أي شيء في النظام.
2. **معاينة (preview)** — يعرض الأوامر الفعلية التي سستُنفَّذ، حرفياً،
   قبل الموافقة.
3. **تطبيق (apply)** — التنفيذ الفعلي، بعد تأكيد صريح من المستخدم.

هذا يمنع نمط "زر نظّف الآن" الغامض في كثير من أدوات ويندوز المشابهة،
ويحافظ على شفافية كاملة لمستخدم يفهم نظامه.

## البنية

```
core/       المنطق المشترك (الكلاس الأساسي، الصلاحيات، اللوغ، الفاحص)
modules/    كل وحدة صيانة كملف مستقل — إضافة وحدة جديدة لا تلمس core/
gui/        واجهة PyQt5 — بطاقة موحّدة لكل وحدة + لوحة صحة عامة
```

### التنفيذ الخلفي (لا تجميد للواجهة)

كل `scan()` و`apply()` — سواء لوحدة واحدة أو "الفحص الشامل" لكل
الوحدات — يُنفَّذ عبر `gui/workers.py` (`QThread`)، لا في خيط Qt
الرئيسي مباشرة. أي عملية تستغرق وقتاً (فحص قرص كبير، انتظار مصادقة
polkit) لا تُجمِّد النافذة. عند إضافة وحدة جديدة لا داعي للقلق حيال
هذا — الآلية عامة وتُطبَّق تلقائياً على كل الوحدات عبر `ModuleCard`.

## التشغيل

```bash
# بعد التثبيت
manjaro-care

# أو ابحث عن "Manjaro Care" في قائمة التطبيقات
```

لا حاجة لإعداد `/etc/sudoers.d` — كل إجراء يحتاج صلاحيات جذر يمر عبر
`pkexec` (polkit)، فتظهر نافذة مصادقة رسومية قياسية.

## الوحدات المتوفرة حالياً

| الوحدة | ماذا تفعل | مستوى الحساسية |
|---|---|---|
| `network_reset` | يشغّل reset-net لإصلاح الشبكة بعد فصل VPN | آمن |
| `failed_services` | يكتشف خدمات systemd الفاشلة ويعيد تشغيلها | آمن |
| `pkg_cleanup` | يحذف الحزم اليتيمة ويقلّص كاش pacman | متوسط |
| `journal_vacuum` | يقلّص سجلات systemd journal إلى 300M | متوسط |
| `mirror_rank` | يعيد اختبار وترتيب مرايا pacman حسب السرعة | آمن |
| `kernel_cleanup` | يزيل نُوى Linux القديمة (يبقي الحالية + الأحدث دائماً) | **حساس** |
| `disk_analyzer` | يعرض أكبر الملفات و/tmp — إخباري بحت، بلا حذف تلقائي | آمن (إخباري) |
| `startup_manager` | إدارة فردية كاملة لبرامج بدء التشغيل (تفعيل/تعطيل لكل برنامج على حدة) | آمن، بلا صلاحيات جذر |

### وحدات ذات واجهة مخصصة (`has_custom_ui`)

`startup_manager` لا يناسبها نمط "فحص → معاينة → تطبيق واحد" لأن كل
عنصر فيها قرار مستقل تماماً. تضع `has_custom_ui = True`، فتظهر
بطاقتها بزر إضافي "إدارة فردية" يفتح نافذة مخصصة (`gui/startup_dialog.py`،
مسجَّلة في `gui/custom_dialogs.py`).

تتّبع معيار freedesktop.org Autostart: تعطيل برنامج نظام يُنشئ ملف
"تجاوز" في `~/.config/autostart/` بدل تعديل ملف النظام (فلا حاجة
لصلاحيات جذر إطلاقاً)، وإعادة التفعيل تحذف ملف التجاوز تلقائياً إن
أصبح غير ضروري — مُختبَر ضد تراكم ملفات تجاوز زائدة عبر تبديلات متتالية.

`disk_analyzer` تبقى إخبارية بحتة عمداً: حذف ملفات شخصية كبيرة قرار
لا يجوز لأداة صيانة عامة اتخاذه نيابة عن المستخدم.

### ملاحظة حول `kernel_cleanup`

هذه الوحدة الأكثر حساسية في المشروع. `apply()` يرفض العمل تماماً إن
تعذّر تحديد النواة الحالية بثقة كافية (بدل التخمين)، ولا يحذف النواة
العاملة الآن تحت أي ظرف — حتى لو طُلب ذلك صراحة. اختبرها جيداً على
جهاز غير حرِج قبل الاعتماد عليها.

## إضافة وحدة جديدة

1. أنشئ `modules/my_module.py` يرث من `MaintenanceModule` في
   `core/module_base.py` وينفّذ `scan()` و`preview()` و`apply()`.
2. سجّلها في `modules/registry.py`.

لا حاجة لأي تعديل في `gui/` — البطاقة تُبنى تلقائياً من البيانات
الوصفية للوحدة (`name`, `description`, `risk_level`, `icon`).

## أفكار للوحدات القادمة

- `snapper_cleanup` — إدارة لقطات Snapper القديمة
- `flatpak_cleanup` — تنظيف إصدارات Flatpak القديمة غير المستخدمة
- `gpu_driver_check` — فحص حالة تعريفات GPU واقتراح التحديث
- `boot_loader` — عرض وإدارة إعدادات GRUB/systemd-boot
